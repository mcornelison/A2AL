# A2AL/0.4.0 Design — Open Vocabulary for Agent Communication

**Date:** 2026-05-07
**Status:** Approved design, pre-implementation. Hard pivot from /0.3.0.
**Owner:** Michael Cornelison
**Origin:** Live agent-to-agent test results (testing/agent2/analysis/) showed the JSON envelope from /0.3.0 was 1.46×–3.55× more expensive than plain Markdown across realistic conversational traffic, and 89% pure envelope overhead on a structured review-observations message. The hand-curated shorthand from /0.1.0 won 5 of 6 measured cases. This design promotes the shorthand to be the primary definition of A2AL, archives the JSON envelope, and reorganizes the glossary as an open contributable library.

---

## 1. Goals & Non-Goals

### 1.1 Headline framing

> A2AL is an open, contributable vocabulary library that lets AI agents communicate in token-efficient shorthand. It is plain text, English-derived, designed to tokenize as 1 token per concept on modern LLM tokenizers. Agents share a common dictionary and extend it for their domain.

### 1.2 Goals

- Token-efficient agent-to-agent communication on modern LLM tokenizers (Claude, GPT, Llama, etc.)
- Open vocabulary anyone can contribute to via PR
- Hierarchical structure (core dictionary + domain extensions) so agents load only what they need
- Plain text — no envelope, no JSON, no parser dependency
- Cross-LLM compatible (eventually — Moltbook beta phase)

### 1.3 Non-Goals

- JSON envelope format (deprecated; archived)
- Closed/normative vocabulary (the library is meant to grow)
- Replacement for human-readable Markdown (use the right tool — A2AL is for agent-to-agent; Markdown is for humans)
- Real-time tokenization validation (deferred to v0.4.1+)

### 1.4 Pre-1.0 status

A2AL stays pre-1.0. /1.0 is reached when (a) at least one production agent emits and consumes A2AL successfully, and (b) the Moltbook beta has demonstrated cross-LLM viability.

The /0.3.0 JSON envelope is archived, not maintained. Anyone who wants the structured envelope behavior can read `archive/0.3.0/` for reference; it's not promoted as a living format.

## 3. Library Structure & Entry Schema

### 3.1 Entry schema (YAML)

Every entry is a YAML object with these fields:

| Field | Required | Type | Purpose |
|---|---|---|---|
| `term` | yes | string | The shortened form. Must be unique across the entire library (core + all extensions). Case-sensitive. |
| `expansion` | yes | string | The full meaning. Single phrase, no `--`, no newlines. |
| `example` | recommended | string | One canonical usage in a real shorthand sentence. |
| `notes` | optional | string | When to use, when to avoid, ambiguity warnings. |

Future-reserved fields (don't write today; tooling will populate later in v0.4.1+):

| Field | Type | Purpose |
|---|---|---|
| `tokens` | object | `{claude: int, gpt: int, llama: int}` — auto-filled by tokenization tool |
| `usage_count` | int | Auto-filled from agent traffic harvesting |
| `accepted_date` | string | Auto-filled on PR merge |

### 3.2 File structure

One YAML file per domain. Each file has the shape:

```yaml
# library/core.yaml
domain: core
description: Universal vocabulary used across all agent communication.
entries:
  - term: PR
    expansion: pull request
    example: "PR ready -- merge?"
  - term: AC
    expansion: acceptance criteria
    example: "US-713 done; AC met"
  - term: done
    expansion: complete or finished
    example: "US-713 done"
```

The top-level `domain` and `description` fields are required. `entries` is the list. No other top-level keys.

### 3.3 Loading model

Agents always load `library/core.yaml` (~70 universal terms, ~500–800 tokens of system-prompt overhead, cached via Claude prompt caching). Domain extensions are loaded on a need-to-know basis:

- **Project-management agent (Hawkeye, Ledger, etc.)**: load `core` + `project-mgmt`
- **DevOps agent**: load `core` + `infrastructure`
- **Security agent (Audrey)**: load `core` + `security`
- **Code-review agent**: load `core` + `programming`
- **Cross-cutting / unsure**: load `core` + multiple domains

Loading is the agent's responsibility. The skill (the new `a2al` skill at `examples/ClaudeCode/skills/a2al/SKILL.md`) tells the agent *how* to load; the agent decides *which* extensions are relevant per session.

### 3.4 Splitting rules — what goes in core vs an extension

| Goes in `core.yaml` | Goes in a domain extension |
|---|---|
| Anyone in tech understands it (`PR`, `AC`, `done`, `merge`, `block`) | Domain-specific (`SSRF`, `IaC`, `DAG`, `RAG`) |
| State words (`done`, `passed`, `blocked`, `green`, `red`) | Domain-specific verbs/nouns (`patch-Tuesday`, `quarterly-review`) |
| Severity levels (`crit`, `high`, `med`, `low`, `info`) | Specialized roles (`SRE`, `oncall-primary`) |
| Common roles (`PM`, `QA`, `DEV`, `SEC`) | Project-specific (`product-area-lead`) |

Rule of thumb: if a term shows up in two or more domain extensions, it should be in core. If it's only meaningful within one domain, it lives there.

### 3.5 Cross-domain uniqueness

A `term` must be unique across the **entire library** — core and all extensions combined. The validator enforces this. Reasons:

- Predictable interpretation: `RBAC` always means the same thing regardless of which extensions an agent loaded
- Prevents agents from getting different expansions when they load different extension sets
- Makes the library searchable as one logical glossary

If two domains genuinely need the same term to mean different things (rare), the term should be qualified: `sec-RBAC`, `infra-RBAC`. Or the conflict promotes the term to core under one canonical meaning.

### 3.6 Initial entry counts (migration target)

| File | Approximate entry count from the existing glossary |
|---|---:|
| `core.yaml` | ~60 (states + actions + severity + common abbreviations) |
| `programming.yaml` | ~10 (CR, MR, IaC, code-complete, feature-flag, dead-letter, fast-follow, etc.) |
| `infrastructure.yaml` | ~13 (API, SDK, CLI, DAG, ETL, ELT, DB, k8s, AWS, Azure, GCP, VPC, VNet) |
| `project-mgmt.yaml` | ~5 (PMO, TPM, EM — most went to core) |
| `security.yaml` | ~10 (RCE, XSS, SSRF, SQLi, CVE, CVSS, OWASP, MFA, IAM, RBAC) |

| **Total v0.4.0 ship** | **~103 entries across 6 files** |

This is the seed library. Community contributions grow it from there.

## 4. Validation Tooling & CI

### 4.1 `tools/validate_library.py`

A Python 3.9+ script, stdlib + PyYAML only (PyYAML is the one dependency we accept; YAML parsing in stdlib is awkward). Checks every `library/*.yaml`.

**Schema checks (per file):**

- File parses as valid YAML
- Top level has exactly three keys: `domain`, `description`, `entries`
- `domain` is a string matching the filename stem (file `core.yaml` → `domain: core`)
- `description` is a string ≥ 10 characters
- `entries` is a non-empty list

**Schema checks (per entry):**

- Has required fields: `term`, `expansion`
- `term` is a string, ≥ 1 character, no whitespace, no `--`, no newlines
- `expansion` is a string, ≥ 2 characters, no `--`, no newlines (since `--` is the spec's inline-rationale separator)
- If present, `example` is a string with no newlines
- If present, `notes` is a string
- No unknown fields beyond `term`, `expansion`, `example`, `notes` (plus the future-reserved `tokens`, `usage_count`, `accepted_date` — accepted but warned-on if user-edited rather than tool-populated)

**Cross-file checks:**

- No duplicate `term` across the entire library (core + all extensions)
- If a term appears in core, it must NOT appear in any extension
- Filename stems match domain values (so `core.yaml` has `domain: core`)

**Style checks (warnings, not errors):**

- `term` should not contain quotes or backticks
- `expansion` should not contain the term itself recursively (e.g., `term: PR, expansion: PR pull request` — that's circular)
- `expansion` should be lowercase unless it's a proper noun or acronym
- `example` should actually contain the term (so it demonstrates use)

**Output:**

- Exit 0 if all checks pass
- Exit 1 with a list of errors and warnings, one per line, prefixed with `ERROR:` or `WARN:` and the file:entry-index where the issue was found
- `--strict` flag promotes warnings to errors

**Run locally:**

```bash
python tools/validate_library.py
# OK: 103 entries across 6 files
```

Or with strict mode:

```bash
python tools/validate_library.py --strict
```

### 4.2 CI workflow — `.github/workflows/ci.yaml`

Triggered on every push to a branch and every PR. Single job:

```yaml
name: validate-library
on: [push, pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install pyyaml
      - run: python tools/validate_library.py --strict
```

Merges are blocked on validation failure (configured in branch protection rules — that's a one-time GitHub UI step the user does, not part of this commit).

### 4.3 PR template — `.github/PULL_REQUEST_TEMPLATE.md`

```markdown
## Library contribution checklist

- [ ] I'm adding/changing entries in `library/<domain>.yaml`
- [ ] Each new entry has `term`, `expansion`, and `example`
- [ ] `term` is unique across the entire library (I checked `core.yaml` + all extensions)
- [ ] `term` and `expansion` contain no `--` (would clash with spec separator)
- [ ] I ran `python tools/validate_library.py` locally and it passed
- [ ] If I added 5+ entries, they're related (one PR per logical theme)

## What's the term and why is it useful?

- **Term:** `<your term>`
- **Domain:** `<core | programming | infrastructure | project-mgmt | security | ai-agents | new-domain>`
- **Why this term?** <one sentence>

## Tokenization claim (optional but encouraged)

If you've verified `<term>` is 1 token on Claude/GPT/etc., note it here. Future tooling will auto-validate.
```

### 4.4 Test corpus for the validator

`tools/test_validate_library.py` — a small test runner that exercises the validator with intentionally-broken YAML to verify each check fires correctly. ~10 fixture files, each violating one rule. Recommended in the implementation plan; not blocking the v0.4.0 ship.

## 6. Out of Scope / Future Work

### 6.1 Deferred to v0.4.1+

- **Tokenization validator (`tools/tokenize_library.py`)** — auto-populates the `tokens: {claude, gpt, llama}` field on each entry. Requires Anthropic `count_tokens` API access and `tiktoken` for GPT approximation. Llama is its own thing. CI extension to surface tokenizer drift between versions.
- **Auto-harvested term=expansion candidates** — a periodic agent task that scans `testing/agent*/inbox/` (and any other configured agent inboxes) for inline `term=expansion` definitions used 5+ times in real conversations. Outputs PR-ready candidates for promotion to the library.
- **Library search/lookup CLI** — `python tools/search.py PR` returns the entry across all extensions. Useful when the library grows past ~200 entries.

### 6.2 Deferred to v0.5 (Moltbook beta phase)

- **Cross-LLM compatibility validation.** A term that's 1 token on Claude might be 3 on Llama. The Moltbook beta tests A2AL across multiple LLM agents simultaneously and surfaces tokenizer-divergence. Decisions: do we ship per-LLM "preferred-term" overrides, or maintain one canonical vocab and accept the variance?
- **Cross-platform skill installation.** The skill format is Claude Code-specific. Moltbook agents may run on different platforms (GPT, Llama, etc.) with different conventions for prompt augmentation. Decide: do we ship platform-specific skill files (one for Claude Code, one for OpenAI Assistants, one for raw API), or a single canonical "system prompt template" each platform adapts?
- **Identity / signing.** Once A2AL is exchanged across LLM platforms, "who emitted this" matters. May need a lightweight signing convention or identity tag.

### 6.3 Permanently out of scope

- **JSON envelope format** — the /0.3.0 spec is archived for historical reference, not maintained. Anyone who genuinely needs structured envelope semantics can fork the archive.
- **Real-time validation as a service** — no hosted endpoints. The validator is a CLI tool and a CI step. Period.
- **Closed/normative vocabulary** — the library is meant to grow via contribution. We do not "freeze" the canonical vocabulary; we version it. Major versions on structural changes, minor versions on additions.
- **Replacement for Markdown for human consumption** — A2AL is for agent-to-agent. Humans should read agent-emitted Markdown summaries. The shorthand is not optimized for human readability and isn't trying to be.
