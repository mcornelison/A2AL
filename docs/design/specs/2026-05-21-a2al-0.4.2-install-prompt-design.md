# A2AL/0.4.2 Design — Idempotent Install Prompt

**Date:** 2026-05-21
**Status:** Approved design, pre-implementation.
**Owner:** Michael Cornelison
**Origin:** /0.4.1 shipped a human-readable install guide (`examples/ClaudeCode/README.md`) — fine for a human operator but verbose for an AI agent driving its own install. The 0.4.2 roadmap entry calls for an AI-agent-consumable install prompt that an installing agent can read once and execute end-to-end, with the same flow handling fresh installs and re-syncs. Idempotency means re-running the prompt on an already-installed agent re-syncs files to current and diffs the CLAUDE.md A2AL block against the current sample — buying upgrade behavior for free, without a separate upgrade prompt.

---

## 1. Goals & Non-Goals

### 1.1 Headline framing

> A single Markdown file at `examples/ClaudeCode/INSTALL-PROMPT.md` that an AI agent reads once and follows to install or re-sync A2AL on a host project. Idempotent by construction: re-running it diffs and refreshes; it never duplicates, never requires a separate upgrade flow inside 0.x.x.

### 1.2 Goals

- An AI agent in a fresh Claude Code session can install A2AL with one operator paste.
- The same prompt handles re-sync: file refreshes are deterministic, CLAUDE.md gets a section-by-section diff review.
- Operator interaction is conversational and minimal (≤5 questions on fresh install; 1 confirmation on re-sync).
- The prompt installs *whatever is current on `main`* — no version pinning inside the prompt.
- Free upgrade path inside 0.x.x: spec additions land via re-sync, no separate upgrade prompt needed.

### 1.3 Non-Goals

- Version-checking, manifest writing, or telemetry. Detection is pure file-existence + a regex on CLAUDE.md.
- Backup/rollback. The skill, command, and library files are deterministic mirrors of upstream; CLAUDE.md is the only thing that drifts, and it's diff-reviewed per subsection.
- Recovery from operator hand-edits to the skill or command. Those files have no operator-edited content by design; if the operator changed them, the prompt overwrites and surfaces a note. This is a documented non-supported customization.
- Automated CI testing of the prompt itself. The prompt is chat-driven; testing is scenario-based.
- A dedicated upgrade prompt. Re-sync covers upgrade for all 0.x.x. A dedicated upgrade flow is only revisited if /1.0.0 ships a normative breaking change.

### 1.4 Relationship to 0.4.1

0.4.1's install artifacts stay untouched:
- `examples/ClaudeCode/README.md` — human-readable guide
- `examples/ClaudeCode/CLAUDE-sample.md` — the canonical A2AL block to install/diff against
- `examples/ClaudeCode/skills/a2al/SKILL.md` — the skill to copy
- `examples/ClaudeCode/commands/a2al.md` — the command to copy

0.4.2 adds exactly one new file: `examples/ClaudeCode/INSTALL-PROMPT.md`. The README gains a "For AI agents — automated install" section at the top documenting the two entry points.

## 2. Architecture

### 2.1 Two phases

```
Phase 1 — Decisions (conversational, one-at-a-time)
   q1. Detect: is A2AL already installed?       computed, not asked
   q2. Identity: [AgentName] + [role]?          asked (fresh) / read from CLAUDE.md (resync)
   q3. Peers + inbox?                           asked (fresh) / read from CLAUDE.md (resync)
   q4. Scope: project or user-global?           asked (fresh) / from detection (resync)
   q5. Library: clone-and-point or copy-locally? asked (fresh) / from CLAUDE.md (resync)

Phase 2 — Execution (deterministic, scripted)
   e1. Acquire A2AL repo (clone or git pull)
   e2. Copy skill + command into chosen scope
   e3. Place library files per chosen mode
   e4. Write or update CLAUDE.md A2AL block
   e5. Verify with the 3-prompt smoke test
```

### 2.2 Mode branches

- **Fresh:** Phase 1 runs q2–q5 in order. Phase 2 runs e1–e5 in order.
- **Re-sync:** Phase 1 collapses to a single confirmation question (q1 has computed everything). Phase 2 runs e1–e3 as deterministic re-copies; e4 becomes the diff loop (Section 5); e5 runs as before.

### 2.3 Entry points

Documented in `examples/ClaudeCode/README.md`:

- **Primary — WebFetch:** Operator pastes a 2-line message into a fresh Claude session:
  > Fetch and follow this install prompt:
  > `https://raw.githubusercontent.com/mcornelison/A2AL/main/examples/ClaudeCode/INSTALL-PROMPT.md`
- **Fallback — inline paste:** Operator copies the contents of `INSTALL-PROMPT.md` from GitHub UI and pastes the whole file into the chat. Self-contained; works without WebFetch.

### 2.4 Scope

In scope: the 5 decisions, the 5 execution steps, the diff loop, the verification step.
Out of scope: peer-discovery, agent-identity validation, any inter-agent comms during install, version pinning, manifest creation.

## 3. Phase 1 — Decisions

Each question is a normal conversational prompt to the human operator. One question at a time.

### 3.1 q1 — Detection (computed, not asked)

The agent runs three probes in order:

1. `Read .claude/skills/a2al/SKILL.md` → if it exists, `scope=project`.
2. Else `Read ~/.claude/skills/a2al/SKILL.md` → if it exists, `scope=user-global`.
3. `Grep '^## A2AL/0\.4' CLAUDE.md` → if it matches, `claude_md_has_block=true`.

**Resync mode** when `scope != none` OR `claude_md_has_block=true`. Otherwise **fresh mode**. The agent announces which mode and why before continuing (one line, e.g., "Detected `.claude/skills/a2al/` — running re-sync.").

### 3.2 q2 — Identity (fresh only)

> What's this agent's name and role? Examples: `Hawkeye/QA`, `Byte/DEV`, `Ledger/PM`. Roles are free-form — pick whatever describes the agent's job on this project.

On resync, identity is read from the existing CLAUDE.md Identity subsection.

### 3.3 q3 — Peers (fresh only)

> Will this agent receive A2AL messages from peer agents? If yes, give me the absolute path to its inbox directory (e.g., `/Users/x/code/myproj/offices/qa/inbox`). If no, I'll omit the Inbox/outbox subsection.

On resync, peer/inbox state is read from the existing CLAUDE.md Inbox/outbox subsection (presence or absence).

### 3.4 q4 — Scope (fresh only)

> Project-scoped (`.claude/` in this project) or user-global (`~/.claude/`)?

On resync, scope comes from detection (q1).

### 3.5 q5 — Library location (fresh only)

> Should the library live (A) inside a cloned A2AL repo I keep at a stable path, or (B) copied into `.claude/a2al-library/` for self-containment? Most installs pick A.

On resync, library mode is inferred from the path in the existing CLAUDE.md Library location subsection (path inside an A2AL clone → clone-and-point; path under `.claude/a2al-library/` → copy-locally). If the path matches neither pattern (operator used a custom location), the agent asks the operator which mode applies before proceeding.

### 3.6 Resync condensation

In re-sync mode, the only operator-facing Phase-1 question is one confirmation:

> Detected A2AL installed at `<scope>`, identity `<Name>/<Role>` from CLAUDE.md, library at `<path>`. Proceed with re-sync? (yes / change something)

"Change something" re-opens q2–q5 selectively for the items the operator wants to update.

## 4. Phase 2 — Execution

### 4.1 e1 — Acquire the A2AL repo

**Fresh mode:**

1. If `~/A2AL` does not exist, run `git clone https://github.com/mcornelison/A2AL.git ~/A2AL`.
2. If `~/A2AL` exists but is not a git repo or is not the A2AL remote, prompt the operator for an alternate clone path. Do not overwrite an unknown directory.
3. If it exists and is the A2AL clone, run `git -C ~/A2AL pull --ff-only` (best effort; warn but continue on failure).

**Re-sync mode:**

- Read the library path out of the existing CLAUDE.md Library location subsection (first absolute path on the line).
- If that path's parent is a git repo and the repo is `mcornelison/A2AL`, `git pull --ff-only` it.
- If the operator chose copy-locally originally, the path will be a non-repo `a2al-library/`. The agent re-clones (or re-pulls) to a scratch location to get current library files — see 4.3.

### 4.2 e2 — Copy skill + command into chosen scope

Deterministic file copy from `<clone>/examples/ClaudeCode/` into the chosen `.claude/`:

```
<clone>/examples/ClaudeCode/skills/a2al/SKILL.md  ->  <scope>/.claude/skills/a2al/SKILL.md
<clone>/examples/ClaudeCode/commands/a2al.md      ->  <scope>/.claude/commands/a2al.md
```

Both files overwrite unconditionally. `mkdir -p` the parent dirs first. No file-level diff prompt — these have no operator-edited content. If the operator did hand-edit them, the final summary notes: "If you edited the skill or command directly, your edits were overwritten."

### 4.3 e3 — Place library files (mode-dependent)

- **Clone-and-point:** nothing to do beyond e1; the library lives in the clone.
- **Copy-locally:** copy `<clone>/library/*.yaml` into `<scope>/.claude/a2al-library/`. Overwrite unconditionally.

### 4.4 e4 — Write or update CLAUDE.md A2AL block

**Target CLAUDE.md location:**
- Project scope (q4 = project) → `<project-root>/CLAUDE.md`
- User-global scope (q4 = user-global) → `~/.claude/CLAUDE.md`

**Fresh mode:**

1. If no CLAUDE.md exists at the target location, copy `CLAUDE-sample.md` to it and fill in placeholders from q2/q3/q5 answers.
2. If CLAUDE.md exists at the target location with no A2AL block, extract the A2AL block from `CLAUDE-sample.md` (the H2 matching `^## A2AL/0\.4\.\d+ — ` and everything up to the next H2) and insert it immediately after the first H2 of the existing CLAUDE.md. Fill in placeholders.

**Re-sync mode:** the diff loop, detailed in Section 5.

### 4.5 e5 — Verify

Run the 3-prompt smoke test from the README inside the same session:

1. Count entries in `library/core.yaml` (expect ~77, plus 3 sample terms).
2. Compose a one-message A2AL test from `<Name>/<Role>` to `Agent2`.
3. Confirm `/a2al` is registered.

If any step fails, the agent prints the README's troubleshooting row that matches the failure.

## 5. The Diff Loop (re-sync e4)

The load-bearing piece. Surfaces spec changes the operator should review; hides noise from their own legitimate placeholder fill-ins.

### 5.1 Extract at section granularity

The A2AL block has six H3 subsections (per `CLAUDE-sample.md`):

```
### Identity for A2AL routing headers
### Library location
### When to use A2AL (audience rule)
### Routing header
### Inbox / outbox (only if you have peer agents)
### Reference
```

The agent parses both the existing CLAUDE.md A2AL block and `CLAUDE-sample.md` into a `{subsection_heading -> body_text}` map.

### 5.2 Normalize placeholders before diffing

Without normalization, every install would show every subsection as drifted (operator put real values where the sample has `[bracketed]` placeholders). One-time extraction pass over the operator's existing block:

| Sample placeholder | Read it from operator's CLAUDE.md |
|---|---|
| `[AgentName]`, `[role]` | Identity subsection — the `<Name>/<Role>` token |
| `[/absolute/path/to/A2AL]` | Library location subsection — the absolute path before `/library/` |
| `[/absolute/path/to/your-inbox/]` | Inbox/outbox subsection — the bullet "Your inbox:" path |
| `[Project Name]` | The file's first H1 (if present) |

Then substitute the operator's real values back to their placeholders in the normalized copy. Both sides now use bracketed tokens; only genuine spec drift remains in the diff.

### 5.3 Diff section-by-section

For each H3 subsection:

- **Equal (after collapsing trailing whitespace):** silent. No operator prompt.
- **In sample, not in operator's block:** "new subsection added in 0.4.X — paste in?"
- **In operator's block, not in sample:** "subsection removed from sample — keep yours / drop it?"
- **Both present, normalized bodies differ:** present diff, ask:

```
=== Subsection: ### Routing header ===
Differences (your CLAUDE.md vs current sample):

  - Optional fields: audience=agent|mixed, urgency=low|medium|high|urgent, refs=<id>,<id>, in-reply-to=<id>.
  + Optional fields: audience=agent|mixed, urgency=low|medium|high|urgent, refs=<id>,<id>, in-reply-to=<id>, thread=<id>.

How do you want to handle this?
  1. Take the new version (replace yours with the sample text)
  2. Keep yours (do nothing; you've customized this on purpose)
  3. Merge manually (I'll show both, you tell me what the final text should be)
```

After operator chooses, the agent writes the resolution back to CLAUDE.md, **re-applying the operator's placeholder values** (inverse substitution from 5.2 — so `[AgentName]` becomes `Hawkeye` again before write).

### 5.4 Update the H2 version heading

After the subsection loop, if the heading is `## A2AL/0.4.X …` for a version below the sample's, the agent rewrites the heading line. This is the operator's visible signal that the install is complete.

### 5.5 Summary

End of loop, the agent prints:

```
A2AL re-sync complete.
  Skill, command, library: refreshed.
  CLAUDE.md A2AL block: 0.4.1 -> 0.4.2.
    Identity: unchanged
    Library location: unchanged
    Audience rule: unchanged
    Routing header: TOOK NEW (added thread= field)
    Inbox/outbox: unchanged
    Reference: unchanged
```

### 5.6 What this loop deliberately does NOT do

- No automatic merging. Every substantive difference is operator-confirmed.
- No structural editing outside the A2AL block. The rest of CLAUDE.md is untouched.
- No fuzzy matching of subsection headings. A renamed H3 is treated as removed + new (fail-safe).

## 6. Error Handling

| # | Failure | Behavior |
|---|---|---|
| 1 | `git clone` fails (network, auth, disk) | Stop. Print the git error verbatim and the manual clone command. Do not proceed to Phase 2. |
| 2 | `~/A2AL` exists, is a non-A2AL directory | Stop. Ask operator for an alternate clone path. Do not overwrite. |
| 3 | `git pull --ff-only` fails (local commits in clone, etc.) | Warn but continue. Library files in the clone are still usable. Final summary notes: "your clone is N commits behind upstream — `git pull` manually." |
| 4 | CLAUDE.md exists but has no H2 sections at all | Insert the A2AL block at the top of the file with its own H2. Note: "no existing H2 found — A2AL block placed at the top; you may want to add a project-identity H2 before it." |
| 5 | Diff loop — operator picks "merge manually" but never finishes | Write nothing for that subsection until the operator completes. If the session ends, re-running the install re-fires the diff for any subsection that's still drifted (idempotency falls out of the file state, not from saved progress). |
| 6 | Verify step (e5) fails | Print the README's troubleshooting row that matches. Do not roll back; partial installs are recoverable by re-running. |

**No rollback.** Phase 2's file ops are write-once and deterministic. If the operator hits Ctrl+C mid-install, re-running the prompt picks up cleanly. Backups would only add confusion.

**Not recovered from:**
- Skill or command file hand-edited by operator → silently overwritten (documented).
- Library YAML hand-edited in copy-locally mode → silently overwritten. (Clone-and-point operators have git.)
- Diff loop interrupted mid-subsection → next run re-diffs from the file state (any still-drifted subsection re-fires).

## 7. Testing Strategy

A prompt is text — there's no Python module to unit-test.

### 7.1 Layer 1 — embedded smoke test

Already covered by e5 (count library entries, compose a test A2AL message, confirm `/a2al`). Catches "the install ran but nothing actually works" failures.

### 7.2 Layer 2 — manual scenario walkthroughs

Under `testing/install-prompt-0.4.2/`:

```
testing/install-prompt-0.4.2/
  README.md                                  # what each scenario covers
  scenario-1-fresh-bare.md                   # nothing on disk; full flow
  scenario-2-fresh-existing-claude-md.md     # CLAUDE.md exists, no A2AL block
  scenario-3-resync-no-drift.md              # already installed, sample unchanged
  scenario-4-resync-spec-drift.md            # sample changed; diff loop fires
  scenario-5-resync-operator-edits.md        # operator hand-edited a subsection
  scenario-6-clone-collision.md              # ~/A2AL is something else
```

Each scenario is a Markdown file with: pre-state, operator messages, expected agent behavior, expected post-state. Same pattern as `testing/adoption-test-0.4.1/`.

### 7.3 No CI test for the prompt

The 0.4.2 PR check stays as today: `python tools/validate_library.py` against `library/*.yaml`. The install prompt is a chat-driven artifact, not a library entry — CI has nothing useful to assert about it.
